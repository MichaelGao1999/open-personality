# 阶段口令触发器设计方案

> 目标：将「阶段一、阶段二、阶段三、阶段四、阶段五、当前阶段」设计为类似「存档/恢复」的口令机制，写入母库（AGENTS.md + agent-coding-workflow.md），通过母库分发给所有项目。

---

## 一、设计原则

| 原则 | 说明 |
|------|------|
| **口令 = 启动器，不是一键完成** | 触发后先做状态检查，再说明本阶段目标，等待用户确认后才开始执行 |
| **状态检查前置** | 每个阶段口令必须先验证「前置阶段是否完成」，避免跳步 |
| **单一真相来源** | 口令机制只定义「如何启动」和「红线检查」，具体执行规范引用 `agent-coding-workflow.md` 对应章节 |
| **精确匹配防误触** | 同「存档」机制：去除标点后精确匹配才触发 |
| **与已有机制风格一致** | 复用 AGENTS.md §3.5/3.6/3.7 的格式和表达 |

### §3.8 阶段口令触发器

**触发词**：`阶段一` / `阶段二` / `阶段三` / `阶段四` / `阶段五`（去除标点后精确等于）

**防误触**：
- 消息精确匹配阶段口令 → 进入对应阶段的启动流程
- 消息包含阶段口令但还有其他内容 → 视为正常对话，不触发

**通用启动流程**（所有阶段口令共用）：
```
Step 1: 读取 status.md，确认当前阶段和前置条件
Step 2: 前置检查（见下方各阶段的「前置条件」）
Step 3: 输出「阶段启动摘要」（格式见下方模板）
Step 4: 等待用户输入 y 确认或 n 取消
Step 5: 确认后，按 agent-coding-workflow.md 对应章节执行
```

**阶段启动摘要模板**：
```
【阶段X 启动摘要】
📌 当前阶段：阶段X — [阶段名]
✅ 前置条件：[已满足 / 未满足 + 原因]
📝 将产出：[列出本阶段将产出的文件]
📖 执行规范：agent-coding-workflow.md §[章节编号]
⚠️  红线约束：[引用 RULE-xx / brief.md 中不可复议的决策]

确认后开始执行。（y/n）
```

### 各阶段详情

#### §3.8.1 阶段一（「阶段一」）

| 项 | 内容 |
|----|------|
| **目标** | 把模糊的想法变成清晰的需求文档 |
| **前置条件** | 无（纯新项目起点）；如项目已有文档则提示「已非新项目，是否强制重开？」 |
| **将产出** | `docs/proposal.md` |
| **执行规范** | `agent-coding-workflow.md §阶段一` |
| **关键动作** | 用提问方式确认需求，绝不猜测意图；完成 T-01 ~ T-05 技术需求确认 |
| **完成标志** | `docs/proposal.md` 产出，T-01~T-05 有明确答案 |

#### §3.8.2 阶段二（「阶段二」）

| 项 | 内容 |
|----|------|
| **目标** | 根据需求文档生成概要设计文档 |
| **前置条件** | `docs/proposal.md` 已存在且非占位符；status.md 显示阶段一 ✅ |
| **将产出** | `docs/design.md`、`docs/brief.md`、条件性 `docs/frontend.md`（T-05②）、条件性 `docs/database.md`（T-01②/③） |
| **执行规范** | `agent-coding-workflow.md §阶段二` |
| **关键动作** | 设计前先做反模式检查；每确认一项决策同步大白话记录到 brief.md |
| **完成标志** | `docs/design.md` + `docs/brief.md` 产出，阶段边界 = 会话边界（结束当前会话） |

#### §3.8.3 阶段三（「阶段三」）

| 项 | 内容 |
|----|------|
| **目标** | 为每个模块拆分成最小可执行子任务 |
| **前置条件** | `docs/design.md` + `docs/brief.md` 已存在且非占位符；status.md 显示阶段二 ✅ |
| **将产出** | `docs/tasks/task-{module}.md`（每模块一个文件）、`docs/tasks/task-progress.md`（总进度看板） |
| **执行规范** | `agent-coding-workflow.md §阶段三` |
| **关键动作** | 子任务小到「一个 subAgent 一次能做完」；包含测试点；标注优先级（P1/P2） |
| **完成标志** | 所有模块的 task 文件产出 + task-progress.md 总进度表产出 |

#### §3.8.4 阶段四（「阶段四」）

| 项 | 内容 |
|----|------|
| **目标** | 生成 agent coding 用的主控 prompt |
| **前置条件** | `docs/tasks/task-progress.md` 已存在；**条件性检查**：如存在 `docs/frontend.md`，检查 §10 决策检查点的 TODO 项是否全部确认 |
| **将产出** | `prompt.md`（项目根目录） |
| **执行规范** | `agent-coding-workflow.md §阶段四` |
| **关键动作** | prompt.md 作为启动入口（引用式而非复制式），包含「启动声明（必读文件清单）+ 项目特定信息 + 内嵌关键规则摘要 + 验收标准」 |
| **完成标志** | `prompt.md` 产出，frontend.md §10 TODO 全部清零 |

#### §3.8.5 阶段五（「阶段五」）

| 项 | 内容 |
|----|------|
| **目标** | 按批次逐个实现模块，每轮控制在 subAgent 超时范围内 |
| **前置条件** | `prompt.md` 已存在且非占位符 |
| **将产出** | 各模块源码（`src/`）+ 测试（`tests/`）+ 集成 |
| **执行规范** | `agent-coding-workflow.md §阶段五` |
| **关键动作** | Orchestrator 先读后派；拆成多轮接力，每轮有明确交付物；争议时引用 brief.md 纠偏 |
| **完成标志** | 按 task-progress.md 分批完成，每轮会话只做一件事 |

### §3.9 当前阶段查询（「当前阶段」/「状态」）

**触发词**：`当前阶段` 或 `状态`（去除标点后精确匹配任一）

**标准动作序列**：
1. 读取 `status.md`
2. 按以下格式输出摘要：
```
【当前状态】
📌 所处阶段：阶段X — [阶段名]
✅ 已完成：[列出已完成的阶段 / 文件]
📋 待办：[从 status.md 待办章节取前 3 条]
🔴 阻塞项：[如有]
📁 关键入口文件：[列出 3-5 个核心文件]
🚀 建议下一步：[根据当前阶段给出一句建议]
```
3. 等待用户下一步指令

## 四、RULE 补充

在 AGENTS.md §3「核心约束」表格中追加：

| 规则 ID | 规则内容 | 违反后果 |
|---------|---------|---------|
| **RULE-10** | **阶段口令触发后，必须先做前置条件检查（status.md + 前置文件确认），不通过则禁止启动该阶段** | 跳步导致产出无上下文、接口缺失依赖 |
| **RULE-11** | **阶段口令是启动器而非执行器：口令只负责「检查+说明目标+确认启动」，具体执行细节严格引用 agent-coding-workflow.md 对应章节，禁止在口令逻辑中重写执行规范** | 规则在两处维护导致版本分歧 |
